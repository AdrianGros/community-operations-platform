# DAD-M Project Documentation

Status date: 2026-03-17

## Purpose

This document records how the DAD-M method was used in this repository refactor experiment.

It is intentionally concise but operational:

- what DAD-M means in this project
- how the milestone flow was applied
- which artifacts were produced
- which prompt patterns were actually used
- which measurement points should be collected later

## Method baseline

This project followed the DAD-M structure from the local `dadm-framework` reference:

- bootstrap first
- then milestone-by-milestone execution
- each milestone split into `Discover`, `Apply`, `Deploy`, `Monitor`

Reference sources used during setup:

- `framework/core/bootstrap.md`
- `framework/core/phases.md`
- `framework/core/deliverables.md`

## Project-specific DAD-M setup

The setup sequence used here was:

1. project brief
2. safety boundaries
3. milestone plan
4. target information architecture
5. rolling milestone execution with one detailed next milestone, one rough next milestone, and one fixed fallback milestone

Repository-local bootstrap artifacts:

- [`project-brief.md`](../bootstrap/project-brief.md)
- [`safety-boundaries.md`](../bootstrap/safety-boundaries.md)
- [`milestones.md`](../bootstrap/milestones.md)

## Applied operating rules

This project used the following DAD-M interpretation:

- planning and implementation were separated at milestone level
- each milestone produced written artifacts, not only chat decisions
- `Deploy` was not allowed to invent new architecture casually
- `Monitor` explicitly decided whether a fix milestone was needed
- current state, target state, drift, and corrective planning were all documented separately

## Milestone sequence used in this project

Bootstrap and structure baseline:

- `M1` bootstrap the showcase hub
- `M2` stabilize the canonical project entrypoint

Operational program:

- `M3` best-practice research
- `M4` current-state inventory
- `M5` target-state planning
- `M6` backup
- `M7` repository skeleton
- `M8` replace placeholders with real artifacts
- `M9` hardening review
- `M10` target-vs-current drift analysis
- `M11` corrective planning plus repo-safety guidance
- `M12` corrective implementation
- `M13` planned final comparison
- `M14` planned commit/push preparation

## Deliverable pattern used

For each milestone, the project generally produced:

- one `mX-detailed-plan.md`
- one `mX-phase-report.md`
- one or more domain artifacts created by that milestone

The phase reports followed the DAD-M deliverable pattern:

- Status
- Input Summary
- Artifacts
- Risks and Assumptions
- Next Step

And for `Deploy`:

- Implementation Summary
- Files Changed
- Proofs
- Acceptance Checklist
- Next Steps

## Prompting approach used

The project used one repeated prompt pattern rather than unrelated ad-hoc requests.

Stable prompt structure:

1. define the milestone role and goal
2. set DAD-M rules and safety boundaries
3. define Discover tasks
4. define Apply tasks
5. define Deploy tasks
6. define Monitor tasks
7. define acceptance criteria, risks, expected artifacts, next milestone, and fallback milestone

Prompt artifacts and prompt usage notes are documented in:

- [`dadm-prompt-log.md`](./dadm-prompt-log.md)

## What worked well

- milestone boundaries made the repo refactor reviewable
- writing artifacts to disk reduced chat-history dependency
- drift analysis and corrective planning prevented hidden rework
- fixed fallback milestones reduced uncertainty without overplanning

## Known limitations

- prompts used in chat were not all captured verbatim from the first moment
- metrics were not collected systematically from the start
- some internal docs still reflect the fact that the process matured while the project was already underway

## Recommended next process improvement

For future DAD-M projects, collect prompt, milestone, and validation metrics from day one instead of backfilling them after the main execution flow.
